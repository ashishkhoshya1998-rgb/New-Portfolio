## Overview

At Clear, we faced a growing design crisis: fragmented systems across Finance Cloud and Supply Chain Cloud, engineers hardcoding UI because they couldn't trust Figma, and 700+ button variants with zero docs. I led Mint V8 from scratch — inventing PCS Logic that reduced component variants by over 80% and achieved 100% team adoption in 6 weeks.

I led the entire initiative from ideation to implementation. Conducted research on token systems and atomic design, created the foundational token structure, designed core components, authored PCS logic, collaborated with developers to align system in Figma and code, and trained cross-functional teams.

## Phase 1 — Foundation: Tokens Before Everything

I had never led a full design system build. I started by studying Material Design, Carbon, and atomic design principles. One insight stood out: **everything scalable starts with a token system.** Tokens abstract the complexity of visual attributes into manageable, code-friendly constants.

The team wanted quick wins — ship some components now. But without tokens, we'd be painting over the same cracks. "Paint the walls now, or fix the plumbing first." A 3-week token foundation was approved. Every component built after would be theme-ready by default.

### Four Token Categories

**Typography Tokens** — Type scale, font weights, line heights, letter spacing — semantically named by usage context (Display L, Body M, Label S, etc.)

**Spacing Tokens** — Base-4 scale: 4px, 8px, 16px, 24px... Tokenised as S0–S10. Kept spacing consistent across all surfaces.

**Colour Tokens** — Functional names (primary, on-primary, container) not appearance names (blue, grey). Tokens mapped to different values per theme.

**Radius, Elevation & Shadows** — Standardised corner radii (radius/sm, radius/md) and elevation levels — unifying UI depth and shape across products.

## Phase 2 — Components: Atomic Design

We followed an atomic design approach to structure the system into three logical layers. This ensured we never rebuilt the wheel — every molecule and organism used the same atomic base.

- **Atoms** — Tokens, Icons, Typography, Colors — the foundational pieces
- **Molecules** — Buttons, Input fields, Dropdowns — interactive components made of atoms
- **Organisms** — Forms, Modals, Toolbars, Tables — complex structures composed of molecules

We also noticed a visual imbalance when icons appeared too bold or light compared to text. We created **dynamic icon sets** — each icon in multiple visual weights (body, label) — so designers could switch based on component prominence. This brought significant improvement to visual rhythm and hierarchy.

![Dynamic icons — multiple weights per icon, switched by context and prominence](/cs/mint-icons.gif)

## The Breakthrough — Discovering PCS Logic

As we built molecules, we hit a wall — component bloat. 760+ button variants. Engineers still hardcoding. Then a whiteboard session revealed a powerful pattern: **nearly every component = Prefix + Content + Suffix.**

How constrained should each P, C, S slot be? Open means anything goes, fixed means no flexibility. I chose **constrained per component type** — flexibility within guardrails. Buttons: prefix=icons only, content=text, suffix=icons/counters. This prevented misuse while enabling every real-world variant.

## Theming — Two Themes, Zero Component Duplication

We created two core themes — one for Clear Finance Cloud (CFC) and one for Clear Supply Chain (CSC). They differed only in primary colour shades. Thanks to token-first architecture, this required no component changes — just token value swaps.

- **Clear Finance Cloud:** Blue 600 — `mint.cfc.sys.color.primary` → #156CEF
- **Clear Supply Chain:** Purple 600 — `mint.csc.sys.color.primary` → #9333EA

![Live theming: CFC (blue) to CSC (purple) — zero component changes, only token value swaps](/cs/mint-theming.gif)

## Adoption — Bridging Designers and Engineers

We didn't have a dedicated design systems engineering team. So I built adoption as a design problem, not a mandate.

<ProcessStep number="01" title="Weekly PCS Workshops">
  45-min sessions every Thursday. After 4 weeks, all designers were self-sufficient. Engineers attended too — they started contributing to the system.
</ProcessStep>

<ProcessStep number="02" title="Figma Playground">
  A shared experimentation space with interactive PCS components. Designers could explore variants inline with live feedback.
</ProcessStep>

<ProcessStep number="03" title="Token Sync Pipeline">
  Tokens documented in Figma, exported to JSON, implemented by devs as CSS variables. Planned: Style Dictionary → automated Storybook integration.
</ProcessStep>

<ProcessStep number="04" title="Gradual Deprecation">
  2–3 screens per sprint migrated to Mint V8. By week 4, teams were voluntarily migrating their own screens without being asked.
</ProcessStep>

The Design Manager wanted immediate system-wide deprecation of old components. I pushed back — mandates create resistance. Each sprint that proved PCS logic's value converted skeptics into advocates. By week 6, adoption was 100% — because the team **chose** it.

## Looking Ahead

**Pattern Standardisation Across UX** — Auditing and standardising all recurring interaction patterns — error handling, feedback messages, file imports, confirmation flows, and empty states — with consistent UX behaviours and micro-interactions.

**Developer Documentation Portal** — Building a robust Storybook documentation portal as a single source of truth for developers, PMs, and designers.

**Motion & Interaction Tokens** — Adding tokens for micro-interactions, animations, and transition timings to ensure consistency in motion design throughout the platform.

**Automated Token Pipeline** — Style Dictionary for token export to JSON, CSS, SCSS, and JS. Figma plugins for PCS toggling. GitHub automation for token updates.

## Reflection

> "Design systems aren't about one perfect UI kit. They're about building a foundation that scales, adapts, and empowers everyone involved."

**Tokens First, Always.** Everything scalable flows from a token system. Components built without tokens are houses built without foundations.

**Structure > Style.** PCS abstracts structure, not just visuals. It's a thinking model as much as a design pattern — and that's what made it transferable to engineers.

**Adoption Is a Design Problem.** Workshops and playgrounds beat mandates every time. Make the new system easier to use than the old one — and the team will choose it themselves.
